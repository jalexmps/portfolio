Context

Twelve tools. Zero cohesion.

NOS Inovação teams relied on multiple disconnected BackOffice tools to manage operations — each built independently, each working differently, none working well together. Support agents juggled separate systems for every task: different credentials, different interfaces, different logic. There was no unified access, no shared patterns, and no design consistency across any of them. CARBON was built to fix all of it.

One platform. One access point. One design system.

The Problem

Multiple tools. Zero shared logic.

Internal operations teams navigated a patchwork of disconnected systems — each BackOffice built at a different time, with different conventions, different access requirements, and no design consistency. There was no single entry point and no shared interaction patterns across the entire ecosystem.

The result was slow, error-prone work. Agents relied on Excel spreadsheets to bridge gaps between tools. Simple tasks required switching between multiple systems and manually copying data. When anything changed mid-process, it often meant starting over.

"Operations were slow, manual, and error-prone — not because of the people, but because of the tools."

Key symptoms

  • ↓ Multiple BackOffices with different logic — no shared patterns or conventions
  • ↓ No unified access — separate credentials and entry points for every system
  • ↓ Inconsistent UI patterns — every tool looked and behaved differently
  • ↓ Heavy reliance on Excel to fill gaps between disconnected systems
  • ↓ Fragmented workflows — multi-step tasks split across unconnected tools

The Challenge

Designed solo. With zero documentation.

Every constraint that could make a design project harder was present simultaneously — no documentation, no product ownership, no existing design system, and a 20:1 developer-to-designer ratio across 12+ concurrent workstreams.

No documentation or ownership

There were no product specs, no requirement documents, and no clear ownership for most BackOffices. Every system had to be understood from scratch before it could be redesigned.

Raw API responses only

Product requirements had to be mapped directly from raw JSON API responses. Understanding what data existed — and what it meant for users — required close, ongoing collaboration with engineering at every stage.

20 developers. 1 designer.

As the sole designer, I was responsible for every BackOffice across every development workstream simultaneously — building and maintaining a design system that could scale far faster than any individual could alone.

My Role

Solo designer across the entire platform

Sole Product Designer across all of CARBON — responsible for every BackOffice, every component, and every design decision. From building the design system from scratch to designing complex operational workflows, the entire design function was a team of one.

Process

How I approached the problem

01

Shadow Research

Spent several days sitting with stakeholders — observing workflows, noting friction, and building a firsthand understanding of how operations actually ran.

02

Stakeholder Interviews

Structured sessions with team leads across departments — mapping priorities and the specific operational logic each BackOffice needed to support.

03

Requirements from JSON

04

Structured 12+ BackOffices into one coherent system — defining shared navigation, access patterns, and the IA that would make everything feel like a single product.

05

Design & System

Built the design system and all BackOffice UX/UI in parallel — each component shared across workstreams as soon as it was ready, keeping pace with 20 active development streams.

Deep Dive

VIP Carbon — the case within the case

CARBON unified 12+ BackOffices. VIP Carbon shows what that unification meant in practice — taking one of the most complex operational workflows and collapsing it into a single, guided experience.

The VIP team handles calls from NOS's most sensitive clients — high-value accounts where every interaction carries significant weight. Before CARBON, a single support call required agents to navigate 3 separate tools, manually copying data between them at each step.

"If someone calls with an issue, I search their NIF in one tool, copy the ID into another to get their history, check a third tool to book a technician visit, then log it in our own BO. If anything changes — missed visit, wrong address — I have to start almost from scratch." — VIP Support Agent

Reflection

What this project taught me

Designing under ambiguity is a skill

With no documentation and raw API responses as the only source of truth, I had to build comfort with uncertainty. Every decision was made with incomplete information — learning to move forward confidently despite that was as valuable as any design method I know.

Designed Once, Used Everywhere

Covering 12+ BackOffices solo was only possible because of the design system. Every component designed once became a component available everywhere, immediately. Without systems thinking from day one, the project would have been impossible at this scale.

A designer is a partner, not a service

Working alongside 20 developers — embedded in real workflows and stakeholder decisions — showed me that design's greatest leverage isn't in the pixels. It's in being present at the right conversations and shaping the right decisions before a single screen is ever drawn.
